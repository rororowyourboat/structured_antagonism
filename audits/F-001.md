# F-001: Verification report misrepresents results

**Layer:** 5 (Cross-Layer Consistency)
**Severity:** Critical
**Status:** Open
**Date raised:** 2026-03-29

---

## Finding

The saved artifact `models/gan/artifacts/verification.txt` claims:

> System: Generative Adversarial Network
> Checks: 10/10 passed

and lists only G-series checks (G-001 through G-006, producing 10 individual findings).

Live verification against the same model using the full OGS check suite (`ALL_CHECKS`) produces **23 findings: 22 passed, 1 FAILED:**

```
FAIL S-005 (warning): Decision game 'Discriminator': Y='Classification'
  — no incoming contravariant flow (no utility feedback)
```

The full check suite includes:

| Suite | Checks | Findings produced | All passed? |
|-------|--------|-------------------|-------------|
| GDS generic (G-series) | G-001 through G-006 | 10 | Yes |
| OGS type (T-series) | T-003, T-004, T-005 | 8 | Yes |
| OGS structural (S-series) | S-002, S-003, S-005, S-006, S-007 | 5 | **No** (S-005 fails) |

The `verification.txt` artifact selectively reports only the GDS generic checks and omits the entire OGS-specific check suite. The user's summary ("10/10 GDS verification checks passed") repeats this.

The S-005 failure identifies a genuine modeling gap: the Discriminator's `backward_in` channel (`Ground Truth Label`) has no incoming wiring. This is the external signal that, in the actual GAN, provides the discriminator with ground truth about whether a sample is real or fake. See related finding F-002.

## Why this is Critical

1. **Self-referential failure.** This project studies how information asymmetry between evaluator and generator causes system degeneration. A verification report that selectively presents passing checks while omitting a failing check is an instance of the failure mode the project describes.

2. **Downstream trust.** The RDF export, the Mermaid diagram, and the structure report are all treated as validated artifacts because "10/10 checks passed." If the full suite was not run, these artifacts carry less assurance than claimed.

3. **Reproducibility.** A reader who runs `ogs verify` against the compiled IR will see 23 findings, not 10. The discrepancy between saved artifact and reproducible result undermines confidence in the methodology.

## Reproduction steps

```python
from ogs.verification.engine import verify
from models.gan.model import build_ir

ir = build_ir()
report = verify(ir)

passed = [f for f in report.findings if f.passed]
failed = [f for f in report.findings if not f.passed]

print(f"Total: {len(report.findings)}, Passed: {len(passed)}, Failed: {len(failed)}")
for f in failed:
    print(f"FAIL {f.check_id} ({f.severity.value}): {f.message}")
```

Output:

```
Total: 23, Passed: 22, Failed: 1
FAIL S-005 (warning): Decision game 'Discriminator': Y='Classification' — no incoming contravariant flow (no utility feedback)
```

## Falsification condition

This finding would be wrong if:

- The `verification.txt` was generated by a version of the OGS library that did not include S-series checks (version mismatch, not selective reporting), OR
- S-005 is a documented false positive for feedback-loop topologies where one player's utility comes from an external environment rather than from another player in the composition, OR
- The `ogs` CLI `verify` command by default runs only G-series checks and the "10/10" accurately reflects the default run

## Recommended action [Prescriptive]

1. Re-run verification with the full check suite and update `verification.txt` to reflect all 23 findings.
2. Address the S-005 failure: either wire the Discriminator's Ground Truth Label as a `PatternInput`, or document why S-005 is a false positive for this topology and suppress it with an explicit annotation.
3. If the original verification was run with a limited check set, document which check set was used and why.
